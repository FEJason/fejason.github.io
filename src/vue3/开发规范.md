
# 前端统一编码与组件开发规范（Vue 3 + TypeScript）

**发布日期：2026 年 5 月 27 日**
**适用对象：前端开发团队**
**适用场景：Vue 3 + TypeScript 项目开发**

## 规范目标

为提升代码一致性、可维护性和团队协作效率，新项目开发过程中请严格遵守以下编码规范。

---

# 一、基础命名规范

## 1. 文件、目录、CSS 类名

统一使用 **kebab-case（中划线命名）**。

适用于：

* 文件名
* 文件夹名
* 路由目录
* CSS Class
* 工具函数文件
* composables 目录（Vue3 可复用的组合式函数目录，在React 中通常是 hooks）

```bash
user-profile.vue
api/user-service.ts
composables/use-device.ts
utils/format-date.ts
```

### CSS 类名示例

```html
<div class="profile-card"></div>
```

### BEM 命名规范（不强制）

```html
<div class="el-card is-always-shadow" style="max-width: 480px">
  <div class="el-card__header">
    <div class="card-header"><span>Card name</span></div>
  </div>
  <div class="el-card__body">
    <p class="text item">List item 1</p>
    <p class="text item">List item 2</p>
    <p class="text item">List item 3</p>
    <p class="text item">List item 4</p>
  </div>
  <div class="el-card__footer">Footer content</div>
</div>
```

---

## 2. 自定义组件与 Nuxt 内置组件

统一使用 **PascalCase（大驼峰命名）**。

### 使用规范

* 有子内容时使用成对标签
* 无子内容时使用自闭合标签

```vue
<NuxtLink to="/about">关于</NuxtLink>

<BaseUserAvatar />

<DataTable :data="list" />
```

包括：

* 自定义组件
* `<NuxtLayout>`
* `<NuxtPage>`
* `<ClientOnly>`
* `<NuxtLink>` 等 Nuxt 内置组件

---

## 3. 第三方 UI 组件

统一使用 **kebab-case（中划线命名）**。

适用于：

* Element Plus
* Vant
* Ant Design Vue

```vue
<el-button type="primary">
  确定
</el-button>

<van-cell title="设置" is-link />
```

以上可以在 ESLint 中配置 
```json
// 强制模板中组件名称为大驼峰（PascalCase），并支持 --fix 自动修复
'vue/component-name-in-template-casing': [
  'error', // 违规时标红
  'PascalCase', // 强制大驼峰命名
  {
    registeredComponentsOnly: false, // 检查所有标签，不管是否已注册
    ignores: ['/^el-/', '/^vant-/'], // 忽略匹配的标签名
  },
],
```

---

## 4. 函数、Props、变量、事件

统一使用 **camelCase（小驼峰命名）**。

### 函数

```ts
const handleSubmit = () => {};
```

### Props

```ts
defineProps<{
  userName: string;
  isActive: boolean;
}>();
```

### Emits

```ts
const emit = defineEmits<{
  (e: 'update:modelValue', val: string): void;
  (e: 'submitForm'): void;
}>();
```

Props 定义时使用小驼峰，在父组件中使用 kebab-case，ESLint 中已配置保存自动格化。

---

## 5. 类、接口、类型

统一使用 **PascalCase（大驼峰命名）**。

```ts
class UserManager {}

interface User {
  id: number;
  name: string;
}

type ApiResponse<T> = {
  data: T;
  code: number;
};
```

---

## 6. 常量

统一使用 **SCREAMING_SNAKE_CASE（全大写 + 下划线）**。

```ts
const MAX_RETRY_COUNT = 3;
const API_TIMEOUT_MS = 10000;
const DEFAULT_THEME = 'light';
```

适用于：

* 配置项
* 枚举值
* 魔法值
* 不可变常量

---

## 7. 补充建议

### 保持 `<script setup lang="ts">` 简洁

可复用逻辑、复杂业务逻辑应抽离至：

* `composables/`  或者 `hooks`  # 可复用且有状态逻辑（如响应式数据、生命周期、副作用等）
* `utils/`                      # 封装 纯函数、无状态、与框架无关 的通用工具函数

避免单个组件承担过多职责。

### 工程化规范

建议统一接入：

* ESLint       # 代码质量检查
* Prettier     # 代码格式化
* Husky        # Git 提交检查钩子（不符合规范无法提交，根据提示修改代码后才能提交）
* lint-staged  # 运行时检查

实现：

* 自动格式化
* 提交前校验
* 代码质量检查

# 二、组件封装规范

在 Vue 3（`<script setup>` + TypeScript）项目中，组件封装质量直接影响：

* 代码可维护性
* 团队协作效率
* 组件复用能力
* 长期技术债务控制

---

## 1. 总体原则

### 单一职责

一个组件只负责一件事情。

### 高内聚、低耦合

内部逻辑封闭，对外接口清晰，不要在组件内部写太多的根据参数判断的 if else 逻辑。

### 显式优于隐式

所有输入、输出都必须声明。

### 类型安全优先

全面使用 TypeScript 定义：

* Props
* Emits
* Slots（可选）

---

# 三、Props 规范

## 1. 命名规范

统一使用 **camelCase（小驼峰）**。

```vue
<!-- 父组件 -->
<UserCard user-name="Alice" is-active />

<!-- 子组件 -->
<script setup lang="ts">
defineProps<{
  userName: string;
  isActive: boolean;
}>();
</script>
```

---

## 2. 必须声明类型

```ts
interface Props {
  id: number;

  size?: 'small' | 'medium' | 'large';

  config?: {
    timeout: number;
    retry: boolean;
  };
}

const props = withDefaults(defineProps<Props>(), {
  size: 'medium',
  config: () => ({
    timeout: 5000,
    retry: true
  })
});
```

---

## 3. 不要直接修改 Props

Props 应始终视为只读数据。

```ts
const localValue = ref(props.modelValue);
```

或：

```ts
const displayName = computed(
  () => props.user?.name || '匿名'
);
```

---

# 四、事件（Emits）规范

## 1. 命名规范

统一使用 **camelCase**，推荐动词开头。

```ts
const emit = defineEmits<{
  (e: 'update:modelValue', value: string): void;
  (e: 'submitForm'): void;
  (e: 'close'): void;
}>();
```

---

## 2. 遵循 Vue 事件命名惯例

### v-model 同步

```ts
update:modelValue
```
之前的写法显得冗长得多。然而，这样写有助于理解其底层机制。

从 Vue 3.4 开始，推荐的实现方式是使用 defineModel() 宏：
```ts
<script setup>
const model = defineModel()

function update() {
  model.value++
}
</script>

<template>
  <div>Parent bound v-model is: {{ model }}</div>
  <button @click="update">Increment</button>
</template>
```

### 原生语义事件

```ts
click
change
input
```

### 业务事件

```ts
submitOrder
deleteItem
rowClick
```

模板中可使用 kebab-case。

---

## 3. 避免直接传递 DOM Event

推荐：

```ts
emit('rowClick', {
  id: row.id,
  index: rowIndex
});
```

避免：

```ts
emit('rowClick', event);
```

这样能够降低父组件与 DOM 的耦合度。

---

# 五、插槽（Slots）规范

## 1. 插槽命名

统一使用 **camelCase**。

### 子组件

```vue
<template>
  <div>
    <header>
      <slot name="header">默认标题</slot>
    </header>

    <main>
      <slot />
    </main>

    <footer>
      <slot name="actions" />
    </footer>
  </div>
</template>
```

### 父组件

```vue
<CustomCard>
  <template #header>
    我的卡片
  </template>

  <p>内容区域</p>

  <template #actions>
    <button>操作</button>
  </template>
</CustomCard>
```

---

## 2. 作用域插槽

### 子组件

```vue
<slot
  name="item"
  :user="user"
  :index="index"
/>
```

### 父组件

```vue
<template #item="{ user, index }">
  <div>
    {{ index }}. {{ user.name }}
  </div>
</template>
```

---

## 3. 避免过度使用插槽

插槽用于：

✅ 内容定制

不建议用于：

❌ 复杂业务逻辑注入

如果插槽中存在大量逻辑，应考虑：

* 拆分子组件
* 使用 Props 配置

| 项目        | 规范               |
| --------- | ---------------- |
| 文件名       | user-avatar.vue   |
| 组件名称      | 基于文件名自动生成        |
| 使用方式      | `<UserAvatar />` |

| Emits     | camelCase        |
| CSS Class | kebab-case       |

## Nuxt 项目目录示例

页面级组件建议与业务页面解耦。

```bash
components/
├── base/
│   ├── title.vue
│   └── images.vue
└── views/
    └── about/
        └── avatar.vue
```

在服务端项目中，页面组件放在 components/views/ 对应目录下，是为了防止 pages 目录自动生成垃圾路由（单页面应用可以将组件放置在 pages 目录中）

---

# 七、其他最佳实践

## 1. 提供默认内容

```vue
<slot>
  <span class="empty">
    暂无数据
  </span>
</slot>
```

---

## 2. 避免混用 v-model 与复杂事件

建议：

* 单值双向绑定 → `v-model`
* 多值或复杂业务 → 独立事件

---

## 3. 使用 defineExpose 显式暴露方法

### 子组件

```ts
const focusInput = () => {
  inputRef.value?.focus();
};

defineExpose({
  focusInput
});
```

### 父组件

```ts
const cardRef = ref();

cardRef.value?.focusInput();
```

---

## 4. 公共组件添加注释 JSDoc

所有的页面、组件  script  第一行应该明确说明页面用户、组件用途

```ts
<script lang="ts" setup>
/**
 * 用户头像组件
 *
 * @param {string} userId 用户ID（必填）
 * @param {'circle' | 'square'} shape 头像形状
 */
defineProps<{
  userId: string;
  shape?: 'circle' | 'square';
}>();
</script>
```

# 八、反模式（Avoid）

| ❌ 不推荐             | ✅ 推荐                    |
| ----------------- | ----------------------- |
| 直接修改 Props        | 使用 computed 或 ref 派生    |
| `this.$parent` 通信 | Props + Emits           |
| 插槽承载大量业务逻辑        | 拆分独立组件                  |
| 非必要直接操作 DOM       | 使用 Vue 响应式能力            |
| 传递原始 DOM Event    | 传递结构化业务数据               |

---

# 九、组件封装 Checklist

开发公共组件前，请确认：

- Props 使用 TypeScript 显式定义
- 必要 Props 已提供默认值
- 不直接修改 Props
- Emits 命名清晰且语义明确
- Slots 提供合理默认内容
- 文件夹、文件名采用 kebab-case
- CSS Class 使用 kebab-case/BEM
- 暴露 API 使用 defineExpose
- 公共组件具备 JSDoc 注释
- 组件职责单一，可独立复用

## 总结

统一的编码规范不仅能提升代码可读性，还能降低团队协作成本、减少维护难度，并提升项目的长期可扩展性。

我们去看 Element-Plus、Vant 和 Ant Design 的代码，就会发现，它遵循了非常严格的编码规范，他们的代码方便所有开发者阅读，也方便团队成员维护。

**目标：让代码“一眼可读”，让组件“可复用、可维护、可扩展”，不要因为代码规范问题推扯，让代码真正成为团队资产，而非技术债务。**
